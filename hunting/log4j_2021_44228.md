# Hunt — Apache Log4Shell (CVE-2021-44228)

> Look for `${jndi:` in any HTTP request field on internet-facing logs, including the common obfuscation variants. Pivot from any hit to the source IP's full request history.

## Detection key

The vulnerability triggers whenever Log4j logs a string containing `${jndi:LDAP://...}` (or `LDAPS://`, `RMI://`, `DNS://`, `IIOP://`). The string can arrive in **any field** that ends up being logged — most commonly the User-Agent header, but also URL parameters, X-Forwarded-For, Referer, and body fields.

Obfuscation variants seen in the wild:

```
${jndi:ldap://...}                   ← canonical
${${lower:j}ndi:ldap://...}          ← lowercase escape
${${upper:j}ndi:ldap://...}          ← uppercase escape
${${::-j}ndi:ldap://...}             ← default-value escape
${jnd${::-i}:ldap://...}             ← split-character escape
${jndi:${lower:l}dap://...}          ← scheme escape
```

All variants normalise to `jndi:.*?ldap` after case-insensitive matching. The regex `(?i)jndi[\${}:.-]*?(ldap|ldaps|rmi|dns|iiop)` catches the entire family.

## Splunk SPL

```spl
index=webserver earliest=-30d
| eval all_text = url + " " + http_user_agent + " " + http_referer + " " + http_x_forwarded_for + " " + http_cookie
| where match(all_text, "(?i)jndi[\${}:.\-]*?(ldap|ldaps|rmi|dns|iiop)")
| stats count, dc(uri_path) as distinct_paths, values(http_user_agent) as user_agents by src_ip
| sort -count
| head 100
```

## Microsoft Sentinel KQL

```kusto
let lookback = 30d;
W3CIISLog
| where TimeGenerated > ago(lookback)
| extend all_text = strcat(csUriQuery, " ", csUserAgent, " ", csReferer, " ", csCookie)
| where all_text matches regex @"(?i)jndi[\${}:.\-]*?(ldap|ldaps|rmi|dns|iiop)"
| summarize hits = count(),
            distinct_paths = dcount(csUriStem),
            user_agents = make_set(csUserAgent, 10)
    by cIP
| order by hits desc
| take 100
```

## Elastic ES|QL

```esql
FROM logs-apache.access-* 
| WHERE @timestamp > NOW() - 30 days
| EVAL all_text = CONCAT(url.original, " ", user_agent.original, " ", http.request.referrer)
| WHERE all_text RLIKE "(?i)jndi[\$\\{\\}:.\\-]*?(ldap|ldaps|rmi|dns|iiop)"
| STATS hits = COUNT(*),
        distinct_paths = COUNT_DISTINCT(url.path),
        user_agents = VALUES(user_agent.original)
    BY source.ip
| SORT hits DESC
| LIMIT 100
```

## Post-hunt pivot — was any callback successful?

If the hunt above returns any hits, follow up with a DNS query hunt — Log4Shell sends a callback to attacker-controlled LDAP/DNS, and many tester payloads use a unique random hostname:

```spl
index=dns earliest=-30d
| where match(query, "(?i)[a-z0-9]{8,32}\.(oast|interactsh|burpcollab|dnslog|requestbin|ngrok).*")
| stats count, values(query) as queries by src_ip
| sort -count
```

The presence of `interactsh`, `oast`, `burpcollab`, etc. in DNS is high-confidence evidence that a Log4Shell payload reached server-side and executed the JNDI lookup. If you find these, escalate immediately.
