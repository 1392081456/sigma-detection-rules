# Hunt — Spring4Shell (CVE-2022-22965)

> Look for the `class.module.classLoader.*` parameter pattern in HTTP request bodies and query strings. The exploit re-writes Tomcat's `AccessLogValve` properties to drop a webshell into a controllable filesystem location.

## Detection key

The Spring data-binding bug lets an attacker walk the bean property chain from a request parameter into the Tomcat `AccessLogValve` and overwrite its `pattern` / `directory` / `prefix` / `suffix` / `fileDateFormat` properties. The dead giveaway is **any request parameter whose key starts with `class.module.classLoader.`** — no legitimate Spring application uses that name.

URL-encoded variants:

```
class.module.classLoader.resources...                ← canonical
class%2Emodule%2EclassLoader...                      ← URL-encoded
class.%6D%6Fdule.classLoader...                      ← partial encode
class.module.classLoader.resources.context.parent... ← deeper chain
```

## Splunk SPL

```spl
index=webserver earliest=-60d
| eval body = if(isnotnull(http_body), http_body, "")
| eval all_input = url + " " + body
| where match(all_input, "(?i)class[\.%]?(2E)?module[\.%]?(2E)?class[\.%]?(2E)?loader")
| stats count, values(uri_path) as paths, values(http_method) as methods by src_ip
| sort -count
```

## Microsoft Sentinel KQL

```kusto
let lookback = 60d;
W3CIISLog
| where TimeGenerated > ago(lookback)
| extend all_input = strcat(csUriQuery, " ", csUserAgent)
| where all_input matches regex @"(?i)class[\.%]?(2E)?module[\.%]?(2E)?class[\.%]?(2E)?loader"
| summarize hits = count(), paths = make_set(csUriStem, 10), methods = make_set(csMethod, 5) by cIP
| order by hits desc
```

## Elastic ES|QL

```esql
FROM logs-apache.access-*
| WHERE @timestamp > NOW() - 60 days
| EVAL all_input = CONCAT(url.original, " ", user_agent.original)
| WHERE all_input RLIKE "(?i)class[\\.%]?(2E)?module[\\.%]?(2E)?class[\\.%]?(2E)?loader"
| STATS hits = COUNT(*), paths = VALUES(url.path), methods = VALUES(http.request.method) BY source.ip
| SORT hits DESC
```

## Post-hunt pivot — find the dropped webshell

Spring4Shell exploits typically drop a `.jsp` file into Tomcat's webapp directory. Hunt for new `.jsp` files created after any hit above:

```spl
index=file_events earliest=-60d
| where match(file_path, "\.jsp$")
| where parent_process_name="java" OR process_name="java"
| stats earliest(_time) as first_seen, count by host, file_path
| sort -count
```

A new `.jsp` whose first appearance is **within 5 minutes of a `class.module.classLoader` hit** is high-confidence successful exploitation. The next pivot is to check that JSP file's content for typical webshell patterns (`Runtime.getRuntime().exec`, `request.getParameter("cmd")`).
